# Claude Session — 2026-05-24

## What Was Done This Session

### Logout Bug Fix — "Server error" + 45-second timeout

**Symptom:** `POST /auth/logout` returned "Server error" and took 45 seconds to respond. App running in Kubernetes with MySQL on EC2.

---

### Root Cause Analysis

The bug was in **three files**, not just the controller:

#### 1. `src/middleware/auth.ts` — primary cause of 45s timeout
`verifyToken` ran a DB query on **every** protected request before calling the controller:
```ts
// BEFORE (broken):
db.query("SELECT id FROM token_blacklist WHERE token = ?", [token], ...)
```
The `token_blacklist` table was never created in MySQL. With a stale TCP connection (K8s pod idle → EC2 drops socket), this query hung for 45 seconds before the OS-level TCP timeout fired, then returned "Server error".

#### 2. `src/controllers/authController.ts` — secondary error
`logout` inserted into the same non-existent table:
```ts
// BEFORE (broken):
db.query("INSERT INTO token_blacklist (token) VALUES (?)", [token], ...)
```

#### 3. `src/config/db.ts` — root cause of 45s hang
Used `mysql.createConnection()` — a single persistent TCP socket. In Kubernetes:
- EC2 security groups / network idle timeouts silently kill the socket
- Subsequent queries block indefinitely (no reconnect logic)
- OS TCP timeout fires after ~45 seconds → MySQL error → "Server error"

---

### Fixes Applied

#### `src/config/db.ts` — createConnection → createPool
```ts
// AFTER:
const db = mysql.createPool({
  host: process.env.DB_HOST,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  database: process.env.DB_NAME,
  waitForConnections: true,
  connectionLimit: 10,
  queueLimit: 0,
});
```
Pool checks out a fresh connection per query. Dead connections are discarded and replaced automatically. No more 45s hangs.

#### `src/middleware/auth.ts` — removed DB call, verify JWT inline
```ts
// AFTER: zero DB calls in middleware
export function verifyToken(req, res, next) {
  const token = req.headers.authorization?.split(" ")[1];
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.userId = decoded.id;
    req.userEmail = decoded.email;
    next();
  } catch {
    res.status(401).json({ message: "Invalid or expired token" });
  }
}
```
JWT signature + expiry enforced by `jsonwebtoken` — no table lookup needed.

#### `src/controllers/authController.ts` — stateless logout
```ts
// AFTER:
export const logout = (_req: AuthRequest, res: Response): void => {
  res.status(200).json({ message: "Logged out successfully" });
};
```
The frontend (`api.ts`) removes the token from `localStorage` after this returns. That is the source of truth for auth state in this project.

---

### Key Decision: Why No Token Blacklist?

JWT is stateless by design. For this learning project:
- Token validity = valid signature + not expired
- Logout = client deletes the token
- This is correct for the current architecture (localStorage JWT, `output: "export"` Next.js)

A token blacklist requires a DB table, adds latency to every request, and needs a cleanup job for expired tokens. Worth adding only when httpOnly cookies + server-side sessions are introduced (production upgrade path).
