### Part 1: Docker Containerization for apps/app

Created three files to containerize the Next.js app for Kubernetes deployment:

#### 1. `Dockerfile` (written from scratch)
- Multi-stage build:
  - **Stage 1 (builder):** `node:20-alpine` — installs deps, builds with Turborepo
  - **Stage 2 (runner):** `nginx:1.27-alpine` — serves the static export
- `--platform=linux/amd64` flags added initially, then removed — Dockerfile is now platform-agnostic (specify platform at `docker build` time for Kubernetes cross-builds)
- `npm ci` with all workspace `package.json` files copied first (apps/app + apps/bo) for layer caching and npm workspace resolution
- `COPY apps/app/.env.development ./apps/app/.env.production` before the build — promotes Kubernetes env vars (`NEXT_PUBLIC_API_URL=http://node-auth-app-service:3000`) into the JS bundle at build time
- `npx turbo run build --filter=app` — builds only the app workspace, skips bo
- Copies `apps/app/out/` (Next.js static export output) into nginx

#### 2. `nginx.conf`
- Listens on port 3000
- `try_files $uri $uri/ =404` — handles Next.js static export with `trailingSlash: true` (e.g. `/login/` → `login/index.html`)
- Custom 404 error page (`/404.html`) — uses Next.js generated 404
- Cache headers: `/_next/static/` assets cached immutably; HTML files not cached

#### 3. `.dockerignore`
- Excludes `node_modules`, `**/.next`, `**/out`, `.turbo`, `.git`, `.github`, docs
- Critical exclusion: `**/.env.local` and `**/.env.*.local` — prevents `apps/app/.env.local` (which has `http://localhost:3000`) from being copied and overriding `.env.development` at build time

### Part 2: nginx Default Page Bug — Root Cause Found and Fixed

**Symptom:** Docker build succeeded but nginx showed its default "Welcome to nginx!" page instead of the Next.js app.

**Root cause:** `apps/app/src/app/page.tsx` used `redirect()` from `next/navigation` — a server-side function. With `output: "export"`, Next.js generates purely static HTML files with no server at runtime. Calling `redirect()` during the static export build caused the root page to not be exported correctly, leaving `out/index.html` absent or empty. The `COPY --from=builder /app/apps/app/out` step copied nothing useful, so nginx served its own default `index.html`.

**Fix — `apps/app/src/app/page.tsx`** (changed):
```tsx
// BEFORE (broken — server-side, incompatible with output: "export"):
import { redirect } from "next/navigation";
export default function Home() { redirect("/login"); }

// AFTER (fixed — client-side, runs in browser after static HTML loads):
"use client";
import { useEffect } from "react";
import { useRouter } from "next/navigation";
export default function Home() {
  const router = useRouter();
  useEffect(() => { router.replace("/login"); }, [router]);
  return null;
}
```

**Also added to Dockerfile** (between build and COPY):
```dockerfile
RUN ls -la /app/apps/app/out/
```
This prints the `out/` file listing during `docker build` so you can immediately see if the static export produced files.

## Key Rules to Remember

| Rule | Why |
|------|-----|
| `next start` does NOT work with `output: "export"` | Static export has no Node.js server; nginx is required |
| Never use `redirect()` from `next/navigation` in a Server Component when `output: "export"` is set | It's server-side; no server exists in static export to execute it |
| Use `useRouter().replace()` in a `useEffect` for client-side redirects in static export | Runs in the browser after the static HTML loads |
| `.env.local` MUST be in `.dockerignore` | It overrides `.env.production` and would bake localhost URL into the bundle |
| `NEXT_PUBLIC_*` vars are baked into the JS bundle at `next build` time | Must copy the right env file BEFORE the build step in Dockerfile |

### Part 3: Two Env Files for Local K8s vs AWS EKS

**Problem:** Browser calls `http://node-auth-app-service:3000` → fails because the browser is outside Kubernetes and can't resolve K8s service DNS names.

**Solution:** Two separate env files baked into different Docker builds.

#### Files changed/created:

**`apps/app/.env.development`** (updated):
```
NEXT_PUBLIC_API_URL=http://localhost:3000
```
Browser reaches backend via `kubectl port-forward svc/node-auth-app-service 3000:3000`.

**`apps/app/.env.production`** (created):
```
NEXT_PUBLIC_API_URL=http://node-auth-app-service:3000
```
Pod-to-pod communication inside EKS — K8s DNS resolves the service name.

**`Dockerfile`** (comment updated, logic unchanged):
- Still copies `.env.development` → `.env.production` before `next build`
- For EKS production build: remove that COPY line so `.env.production` is used directly

#### Key insight:
`NEXT_PUBLIC_*` vars are baked into the JS bundle at build time — not at runtime. You MUST build two separate Docker images: one for local K8s, one for EKS. You cannot swap the URL at container start time.

### Part 4: MySQL Env Vars for node-auth-app in Kubernetes

**Context:** node-auth-app backend running in local K8s needs to connect to MySQL on EC2 at `13.204.38.60:3306`.

**Approach:** Kubernetes Secret (sensitive) + ConfigMap (non-sensitive) → injected via `kubectl set env`.

#### Commands:

```bash
# 1. Create Secret for sensitive values
kubectl create secret generic node-auth-app-secret \
  --from-literal=DB_PASSWORD=NodePass@123 \
  --from-literal=JWT_SECRET=siva_secret_key_2026

# 2. Create ConfigMap for non-sensitive values
kubectl create configmap node-auth-app-config \
  --from-literal=DB_HOST=13.204.38.60 \
  --from-literal=DB_USER=nodeuser \
  --from-literal=DB_NAME=nodeauthdb \
  --from-literal=PORT=3000

# 3. Inject both into the deployment (triggers rolling restart)
kubectl set env deployment/node-auth-app \
  --from=secret/node-auth-app-secret \
  --from=configmap/node-auth-app-config

# 4. Verify
kubectl rollout status deployment/node-auth-app
kubectl exec deployment/node-auth-app -- env | grep -E "DB_|JWT_|PORT"
```

#### Update a value later:
```bash
# Re-apply secret with --dry-run patch
kubectl create secret generic node-auth-app-secret \
  --from-literal=DB_PASSWORD=NewPassword \
  --from-literal=JWT_SECRET=siva_secret_key_2026 \
  --dry-run=client -o yaml | kubectl apply -f -

# Restart pods to pick up change
kubectl rollout restart deployment/node-auth-app
```
